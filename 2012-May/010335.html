The ability to allow for complex use cases certainly needs to be preserved.  But I believe that this can co-exist with 
simplifying the default configuration - I think that the more complex the initial configuration the more likely it is to 
discourage potential community members and contributors.

It's not strictly true that all module config is independent - many modules look for the StorageProvider setting in the 
[DatabaseService] section.

Possibly one might be able to use $servicehost or $hgaddress in configuration and replace this with a value defined 
elsewhere.  This isn't completely dissimilar from the ability to use env vars in region config that Dahlia implemented 
way back in 2009, though the value would come from config rather than an env var.

However, I'm not aware of any imminent plans to do this.  As other people have said, the workaround right now is to do a 
whole string search and replace on a config file though I don't think this is the best solution one could possibly have.
